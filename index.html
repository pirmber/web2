Data

Data refers to raw facts and figures, such as numbers, characters, images, or any other output that can be processed by a computer. It has no meaning on its own until processed and interpreted.

Information

Information is processed data that is meaningful and useful for decision-making. It is data that has been organized, structured, or presented in a context that gives it meaning.

Features of Information
  • Accuracy: Information must be correct and reliable.
  • Timeliness: Information should be available when needed.
  • Relevance: It must relate to the purpose at hand.
  • Completeness: Should contain all necessary details.
  • Consistency: Information should be consistent across sources.
Database and Its Purpose

A database is an organized collection of related data. Its purpose is to store, manage, retrieve, and update large amounts of data efficiently.

Terminologies in Database
  • Table: A collection of records in rows and columns.
  • Field: A column in a table; a specific attribute.
  • Record: A row in a table; a single data item.
  • Tuple: Another term for a record.
  • Object: A data entity in object-oriented databases.
  • Key: A field or set of fields used to identify records uniquely.
Data Dictionary

A data dictionary is a centralized repository of metadata, describing the structure, relationships, and constraints of data stored in the database.

Database Management System (DBMS)

Introduction

DBMS is software for creating, managing, and interacting with databases. It provides an interface between applications and the data.

Objectives

  • Provide efficient data storage and retrieval.
  • Ensure data integrity and security.
  • Support concurrent access.

Advantages

  • Data independence
  • Reduced data redundancy
  • Improved data sharing

Disadvantages

  • Complexity and cost
  • Performance overhead
Types of Database Model
  • Hierarchical: Tree-like structure with parent-child relationships.
  • Network: Graph-based structure allowing many-to-many relationships.
  • Relational: Data represented in tables, linked by keys.
  • Entity-Relationship: Conceptual model depicting entities and relationships.
Integrity Constraints and Types
  • Domain Constraint: Valid values for a given attribute.
  • Entity Integrity: Primary key cannot be null.
  • Referential Integrity: Foreign key must match a primary key in another table.
  • Key Constraints: Uniqueness of keys.
Normalization

Introduction

Normalization is the process of organizing data to minimize redundancy and dependency.

Normal Forms

  • 1NF: Eliminate repeating groups.
  • 2NF: Remove partial dependencies.
  • 3NF: Remove transitive dependencies.

Advantages

  • Reduces redundancy
  • Increases data integrity

Disadvantages

  • Can be complex
  • May require more joins
Centralized and Distributed Database

Introduction

Centralized DB stores data at a single location; distributed DB stores data across multiple sites.

Advantages

  • Centralized: Easier control.
  • Distributed: Fault tolerance, local autonomy.

Disadvantages

  • Centralized: Single point of failure.
  • Distributed: Complex management.

Comparison

Distribution improves availability, but adds complexity in synchronization and management.

Database Security

Introduction

Database security protects data from unauthorized access and threats.

Challenges

  • SQL injection
  • Unauthorized access
  • Data breaches

Security Measures

  • Authentication and authorization
  • Encryption
  • Auditing and monitoring

Roles of DBA

  • Backup and recovery management
  • User management
  • Performance tuning
Practical Topics (DDL/DML & SQL Data Types)

DDL and DML Languages

DDL (Data Definition Language) includes commands like CREATE, ALTER, DROP. DML (Data Manipulation Language) includes SELECT, INSERT, UPDATE, DELETE.

UNIT 2: Database Management System — Deep Interactive Module

Core Concepts — organized

DBMS — What it is & why it matters

DBMS is software that stores, retrieves and safeguards organized data. It provides: data models (relational), query languages (SQL), transaction management (ACID), concurrency control, backup & recovery, and security. For any multi-user app (banking, ecommerce, LMS), the DBMS enforces correctness and availability while preventing corruption from concurrent access.

When to use a DBMS: multiple users, large datasets, need for complex queries, strong consistency, or when you require controlled access & auditing.

Data vs Information • Features of good information

Data = raw symbols; Information = processed, contextualized data that supports decisions. Good information is accurate, timely, relevant, complete and consistent across systems. In practice, data governance (data dictionary, validation rules) ensures information quality.

Terminologies, Keys & Data Types
  • Table: rows (tuples) and columns (attributes)
  • Primary Key: unique, not-null identifier
  • Foreign Key: references PK of another table for relationships
  • Common types: CHAR/VARCHAR, INT/TINYINT, FLOAT/DOUBLE/DECIMAL, DATE/DATETIME, TEXT/BLOB, ENUM
CREATE TABLE student (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(150) UNIQUE
);
Integrity Constraints & Normalization

Integrity constraints (domain, entity, referential, unique) keep data valid. Normalization organizes tables to remove redundancy and anomalies. 1NF: atomic values. 2NF: no partial dependencies. 3NF: no transitive dependencies. In practice, many production DBs use a mix of normalized tables and denormalized read-models for performance.

DB Security & DBA duties

Security measures: authentication, least-privilege RBAC, prepared statements to avoid SQL injection, encryption in transit & at rest, and logging. DBAs plan backups, tuning, user roles, indexing strategies and disaster recovery.

SQL JOINs (quick overviews)

Joins combine table rows: INNER (only matching), LEFT (all left + matches), RIGHT (all right + matches), FULL (all rows from both), CROSS (cartesian). Use JOINs whenever you need related data across tables.

-- inner join example
SELECT s.name, c.title
FROM students s
INNER JOIN enrollments e ON s.id = e.student_id
INNER JOIN courses c ON e.course_id = c.id;

📚 Essay Topics

Database normalization is a systematic process of organizing data in a database to reduce redundancy and ensure data integrity. It involves breaking down tables into smaller, more manageable parts while maintaining relationships. Normalization typically progresses through stages (1NF, 2NF, 3NF, BCNF, etc.), each removing specific anomalies like duplication, partial dependency, and transitive dependency.

Distributed databases spread data across multiple servers, often in different locations. This offers improved reliability, faster access for local users, and scalability. For businesses, distributed systems ensure that even if one server fails, the entire database does not become unavailable. However, they also add complexity in synchronization and management.

File systems store data in unstructured or semi-structured files, making searching, security, and concurrent access difficult. On the other hand, database systems are designed to handle large amounts of data efficiently, with support for indexing, queries, security, and transactions. Unlike file systems, DBMS ensures data integrity and supports multiple users simultaneously without corruption.

Data security is crucial in modern databases since sensitive information such as financial records, health data, and personal details are stored digitally. DBMS provides authentication, access control, and encryption features to protect data. Ensuring security prevents unauthorized access, data leaks, and ensures compliance with laws like GDPR and HIPAA.

Keys play a fundamental role in relational databases. Primary keys ensure unique identification of each record, while foreign keys establish relationships between tables. Composite keys can combine multiple attributes to create uniqueness. Proper use of keys ensures relational integrity, prevents duplication, and enables efficient queries using joins.

ER → Schema (example)

-- Students, Courses, Enrollment
CREATE TABLE students(id INT PRIMARY KEY, name VARCHAR(80));
CREATE TABLE courses(id INT PRIMARY KEY, title VARCHAR(120));
CREATE TABLE enrollments(student_id INT, course_id INT, PRIMARY KEY(student_id,course_id),
  FOREIGN KEY (student_id) REFERENCES students(id),
  FOREIGN KEY (course_id) REFERENCES courses(id));
